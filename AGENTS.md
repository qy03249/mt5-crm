# 项目开发约定

## 数据库脚本

- 所有数据库迁移脚本、初始化脚本、种子数据脚本必须包含中文注释。
- 表、字段、索引、初始化数据涉及业务含义时，应写清楚中文说明。
- MySQL 表和字段应尽量使用 `comment` 描述业务含义。
- 资金、审核、MT5 写操作相关字段必须在注释中说明用途，避免后续误用。
- 新增数据库脚本前，先查看 `docs/data-field-map.md`，保证字段命名和页面字段一致。

## 配置原则

- 系统名称、Logo、favicon、语言、入出金规则、支付/收款方式、MT5 参数都应做成后台配置。
- 不允许把可变业务参数写死在代码中。
- `.env` 只用于本地环境，不提交真实密钥或密码。

## 参考系统对齐原则

- 后续功能开发必须以已梳理的参考系统菜单、页面字段和业务逻辑为准。
- 功能清单、菜单层级、页面入口、核心操作流程要和参考系统对齐，不允许脱离参考系统另起一套功能结构。
- 前端视觉风格可以自定义并做得更干净，但不能因为改风格而改变参考系统已有功能逻辑。
- 开发新模块前，先查看 `docs/mt5-crm-feature-inventory.md` 和 `docs/data-field-map.md`，确认菜单、字段、筛选项、按钮和页面行为。

## 交付运行原则

- 每次功能开发或修复完成后，应自动执行必要的测试、静态检查、数据库迁移、构建或编译步骤。
- 功能开发必须前后端一起闭环交付：后端接口完成后，同步完成对应前端页面或交互入口，确保用户可以在浏览器中直接验证。
- 每次交付应以“完整菜单功能”为单位，不把列表、新增、编辑、状态管理等同一菜单内的能力拆成多次让用户验证；除非用户明确要求阶段性查看。
- 前端页面不得直接展示英文枚举编码；下拉选项、状态标签、角色、审核状态等枚举值必须显示对应中文，后端仍可保留稳定英文编码作为存储值。
- 如果项目需要服务运行，应自动启动服务；如果服务已经运行，应更新或重启服务，让最新代码生效。
- 完成后只需要向用户说明访问地址、必要账号和关键验证结果，不要求用户手动执行启动命令。
- 如因网络、端口占用、依赖缺失等外部原因无法启动或更新服务，应明确说明阻塞原因和当前本地状态。
